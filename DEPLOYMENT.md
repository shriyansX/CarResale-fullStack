# Deployment Guide

This project deploys cleanly as three services:

- `car-resale-client`: Vite/React static site
- `car-resale-api`: Express API
- `car-resale-ml`: FastAPI ML service

The repository root includes a Render Blueprint in [render.yaml](../render.yaml). The API now uploads listing images directly to ImageKit, so the deployed app does not depend on local filesystem storage.

## Before You Deploy

1. Push this repository to GitHub, GitLab, or Bitbucket.
2. Create a MongoDB Atlas database and copy its connection string.
3. Create an ImageKit account and collect:
   - `IMAGEKIT_PUBLIC_KEY`
   - `IMAGEKIT_PRIVATE_KEY`
   - `IMAGEKIT_URL_ENDPOINT`
4. If you want payments and outgoing emails to work in production, collect:
   - `STRIPE_SECRET_KEY`
   - `SMTP_HOST`
   - `SMTP_PORT`
   - `SMTP_USER`
   - `SMTP_PASS`
   - `EMAIL_FROM`

## Deploy on Render

1. In the Render dashboard, choose `New -> Blueprint`.
2. Connect the repository root that contains `render.yaml`.
3. Deploy the Blueprint. Render will provision:
   - a static site for `CarResale-fullstack/client`
   - a Node web service for `CarResale-fullstack/server`
   - a Python web service for `CarResale-fullstack/ml_backend`
4. When Render prompts for `sync: false` variables, fill in the server secrets.
5. Wait for all three services to finish deploying.

The Blueprint wires these values automatically:

- `VITE_BASE_URL` points the frontend at the API service's public Render URL
- `CLIENT_URL` points Stripe redirects back to the frontend
- `ML_API_URL` points the API to the ML service over Render's internal host and port

## Environment Variables

For local development, use:

- [client/.env.example](./client/.env.example)
- [server/.env.example](./server/.env.example)

For production on Render:

- Required for core app: `MONGODB_URI`, `IMAGEKIT_PUBLIC_KEY`, `IMAGEKIT_PRIVATE_KEY`, `IMAGEKIT_URL_ENDPOINT`
- Auto-generated by Blueprint: `JWT_SECRET`
- Required for Stripe checkout: `STRIPE_SECRET_KEY`
- Required for password reset, newsletter, and payment emails: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `EMAIL_FROM`

## Health Checks

- API: `/health`
- ML service: `/health`

## After Deploy

1. Open the frontend URL and confirm cars load from the API.
2. Open the API health endpoint and confirm it returns `{"success": true}`.
3. Open the ML health endpoint and confirm the model reports `model_loaded: true`.
4. If you add custom domains later, update `VITE_BASE_URL` and `CLIENT_URL` in Render so they use your custom URLs instead of the default `onrender.com` addresses.

## Optional Seed Data

If you want demo data in production, run the server seed script against your production `MONGODB_URI`:

```powershell
cd CarResale-fullstack/server
$env:MONGODB_URI="your_production_connection_string"
npm run seed
```

Use a separate database for demo data if you do not want seed content mixed with real user data.
